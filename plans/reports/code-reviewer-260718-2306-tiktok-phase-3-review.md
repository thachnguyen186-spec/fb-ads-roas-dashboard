## Medium Priority

### 1. New tab-switcher `layout.tsx` adds an un-budgeted ~40px of height to **both** dashboards, likely producing a page-level scrollbar that wasn't there before
`app/dashboard/layout.tsx` wraps `{children}` in a plain, unstyled `<div>` alongside the `<nav>` pill bar. Both `CampaignHub` and `TiktokCampaignHub` still render their own root as `className="flex flex-col bg-slate-50 min-h-screen"`. Root `<body>` is `min-h-full flex flex-col` (`app/layout.tsx:30`). Box-model consequence: the wrapping div's total content height is now `nav_height (~40px, from `px-6 pt-2` + pill `py-1.5`) + max(page_content, 100vh)`. Since the page's own div enforces `min-height: 100vh` regardless of the nav sibling above it, the wrapping div's natural height becomes `100vh + ~40px`, and `body` (which only has a *minimum* height, not a cap) grows to match — producing a page that is now consistently ~40px taller than the viewport on **every** phase (idle/loading/error/results) of **both** `/dashboard` and `/dashboard/tiktok`, not just TikTok's new tab. Before this phase, the page's own `min-h-screen` div was body's only child, so body was exactly one viewport tall with no scroll.
- This is purely cosmetic (a faint always-present vertical scrollbar / ~40px of dead space), not a data or security issue, and does not fail any Success Criteria bullet — but it is a real, unintended visual regression to the *existing* FB dashboard, which the task asked me to confirm is unaffected by this phase.
- Compounding detail: `tiktok-campaign-hub.tsx`'s results-view table wrapper uses `h-[calc(100vh-300px)]`, deliberately 40px *smaller* than FB's own tuned `h-[calc(100vh-340px)]` (`campaign-hub.tsx:831`) for a layout with fewer chrome elements — but neither constant was adjusted for the new nav bar's height, so both are now likely undersized by roughly the nav bar's height in the results view specifically, on top of the page-level overflow above.
- Not confirmed in an actual browser (no test runner in this repo) — flagging from CSS box-model analysis, worth a 30-second visual smoke test on both tabs.
- **Fix (minimal, no page-file changes needed):** in `layout.tsx`, make the wrapper `<div className="flex flex-col h-screen">`, mark `<nav>` `flex-shrink-0`, and wrap `{children}` in a `<div className="flex-1 overflow-y-auto">`. This contains the existing `min-h-screen` children within a bounded, internally-scrolling region instead of letting them add to the document's total height — no change required in `campaign-hub.tsx`, `tiktok-campaign-hub.tsx`, or the vh-calc constants.

## Low Priority / Informational

- **`tiktok-campaign-hub.tsx` error state has no Settings link.** The not-connected/no-advertisers banners above the "Fetch Data" card use `<Link href="/settings">`, but a 409 "TikTok connection expired — reconnect in Settings." error from a failed fetch renders as plain red text (`errorMsg`) with no clickable link, same pattern FB's hub uses for its own token-error message (`campaign-hub.tsx:713`) — consistent with the existing codebase convention, not a regression, just noting the inconsistency exists in both places if it's ever worth fixing.
- **Search-box placeholder implies a feature that's inert on the TikTok tab.** `FilterBar`'s placeholder text ("Search campaign name… (Enter to lock)") is unconditional on `campaignNameKeywords.length`, but for TikTok that array is always `[]` with no-op handlers, so pressing Enter does nothing visible. Not broken (typing still live-filters via `onCampaignNameChange`), just a minor, pre-existing-by-design UX rough edge from reusing a component built for a feature TikTok doesn't have.
- **Fail-fast batch semantics in `route.ts`.** If any single advertiser's `fetchForAdvertiser` call throws inside a `Promise.all` batch (e.g. TikTok API error for one stale advertiser), the whole request 502s, discarding any already-fetched batches. This exactly mirrors the existing FB route's own `Promise.all(accounts.map(...))` fail-fast behavior (`app/api/campaigns/route.ts:90-95`) and is not something Phase 3's spec asked to change (partial-failure resilience is only scoped to Phase 4's bulk actions, finding #15) — noting it as a shared, pre-existing characteristic rather than a new defect.
